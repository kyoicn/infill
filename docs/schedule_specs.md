# 排班算法详细规格

> 本文档是排班算法的详细设计规格，所有排班相关实现以此为准。  
> 总体设计规格见 [specs.md](specs.md)。

---

## 1. 输入参数

- **排班日期** (`target_date`)：排班起始日
- **开始时间** (`start_time`)：当天几点开始，默认 `08:00`
- **排班时长** (`duration_hours`)：总时长，如 24h、48h、168h（一周）
- **调度策略** (`strategy`)：`"product_first"` 或 `"utilization"`，默认 `"product_first"`（见第 5 节）
- **指定产品** (`target_product_ids`)：可选，用户手动指定本次排班只考虑的产品列表（见第 11 节）
- **富余生产开关** (`surplus_enabled`)：排完订单需求后是否继续利用剩余产能

排班周期 = `start_time` ~ `start_time + duration_hours`，所有任务的**启动时间**和**结束时间**都必须在此周期内。

---

## 2. 操作窗口

- 用户可按星期几配置操作时间窗口（如工作日 08:00-12:00 / 12:30-18:00 / 18:30-23:00）
- 打印任务只能在操作窗口内**启动**，但可以**跨窗口运行**（如 11:45 启动，13:45 结束，中间午休不影响）
- 跨天时自动拼接多天的窗口

---

## 3. 换料时间（changeover）

- 每个任务结束后有固定的换料时间（默认 15 分钟），用于收取成品、更换打印版、启动下一轮
- `printer_available = task_end_time + changeover_minutes`，表示该打印机最早可以**启动**下一个任务的时间
- 寻找下一个可启动时间时，不再额外加 changeover（因为已包含在 `printer_available` 中）

---

## 4. 任务时限约束

- 任务的**启动时间**必须在排班周期内（`start < deadline`）
- 任务的**结束时间**也必须在排班周期内（`end <= deadline`）
- 不允许任务超出排班周期，即使是"跑过夜"也必须在周期内结束
- 超长任务（如 776 分钟、882 分钟）需要足够早启动以确保在周期内完成

> 注意：如果排班时长较短（如 24h），超长任务只能在前半段安排。用户可以延长排班时长（如 168h/一周）来增加灵活性。

---

## 5. 调度策略

系统提供两种调度策略，用户在生成排班时选择。两种策略共享相同的任务池（由需求计算和打印配置选择决定），区别仅在于**任务执行顺序**的优先级评估方式。

### 5.1 策略一：优先凑齐发货（`product_first`）— 默认

**目标**：尽早凑齐完整产品以便组装发货。

调度过程中维护一个"模拟库存"（sim_supply），从初始库存开始，每选中一个任务就加上其产出。选择下一个任务时，动态评估每个候选任务对"凑齐完整产品"的贡献，优先安排能让某个产品最快组装完成的瓶颈组件。

**模拟库存跟踪：**

```
1. 初始化 sim_supply = 当前库存 + 早于排班日的已排班产出
2. 构建产品单元队列：按订单 FIFO 展开所有待处理订单的产品单元
3. 尝试组装：如果初始库存已能组装某些产品，消费库存并标记为"已组装"
4. 每次选中任务后：
   - sim_supply[comp] += 任务产出
   - 重新检查是否有新产品可组装（消费库存并标记）
```

组装时消费库存确保共享组件不会被重复计算；按优先级顺序组装确保高优先级订单优先获得共享组件。

**产品完成度评分：**

对每个候选任务，计算其生产的组件对最相关产品的贡献分数：

```
score = (order_priority, -completion_ratio, bottleneck_ratio, idle, duration)

- order_priority: 该组件对应的最高优先级订单（小=早=优先）
- -completion_ratio: 产品当前完成度的负数（接近完成的产品优先）
- bottleneck_ratio: 该组件的供给比例（低=瓶颈=优先）
- idle: 任务完成后打印机的空闲等待时间（小=优先）
- duration: 任务时长（短=优先，为长间隔保留长任务）
```

完成度 = 产品所有组件中最低的 (sim_supply / BOM需求) 比例。

**效果：**

- 集中火力：接近完成的产品会被优先凑齐，而不是所有产品同时推进一点点
- 瓶颈优先：每次打印产出量少的组件（如上柜 1个/盘）自然获得更多尽早安排的机会
- 订单 FIFO：早期订单的产品优先被凑齐，但不会完全阻塞后续订单
- 自适应窗口：idle 和 duration 作为 tiebreaker，保持窗口利用率优化

### 5.2 策略二：最大化利用率（`utilization`）

**目标**：最大化打印机运转时间，减少空闲。

不维护模拟库存，不考虑产品完成度。纯粹以最小化空闲时间为目标选择任务。

**任务选择规则：**

```
score = (order_priority, idle, duration)

- order_priority: 订单 FIFO 优先级（仍然尊重订单顺序）
- idle: 任务完成后打印机的空闲等待时间（小=优先）
- duration: 任务时长（短=优先，为长间隔保留长任务）
```

**效果：**

- 窗口密集时段：长短任务 idle 都 ≈ 0 → 优先短任务 → 自然保留长任务给后续长间隔
- 临近长间隔时段：短任务结束后有大量空闲，长任务跨间隔空闲少 → 自动选长任务
- 产品可能零散完成，不如"优先凑齐"策略集中

**适用场景：**

不急于发货、想要最大化产能利用的阶段，如囤货或产能评估。

### 5.3 两种策略对比

| | 优先凑齐发货 | 最大化利用率 |
|---|---|---|
| **任务池** | 相同 | 相同 |
| **选择核心** | 产品完成度评分 | 最小空闲等待时间 |
| **产品完成** | 集中完成，尽早可发货 | 零散完成 |
| **打印机利用率** | idle 作为 tiebreaker | idle 作为主要指标 |
| **适用场景** | 日常运营，订单稳定 | 囤货，不急发货 |

## 6. 为什么不做批内分散

早期版本曾强制同一批次内 4 台打印机分配不同组件。实测发现这会严重损害效率：

- 产品 BOM 中各组件的打印效率差异极大（如把手 1 盘出 250 个，上柜 1 盘出 1 个）
- 强制分散给每种组件相同的打印机槽位，但瓶颈组件（上柜、下桌）需要大量槽位
- 结果：非瓶颈组件产量过剩，瓶颈组件严重不足，整体能组装的完整产品很少

当前做法：**完全由任务池比例驱动调度**。需求计算阶段已按正确比例生成任务（50 个上柜任务 vs 1 个把手任务），调度阶段按策略选择任务，让瓶颈组件自然获得足够的打印机时间。

## 7. 富余生产

### 7.1 触发条件

- `surplus_enabled = true`
- 订单需求任务全部排完后仍有剩余产能

### 7.2 富余任务池构建 — 瓶颈算法

1. **只考虑目标产品范围内的产品**（如指定了产品过滤，则只考虑被选中的产品；否则考虑所有有待处理订单的产品）
2. 计算"预计库存" = 当前库存 + 需求任务产出
3. 对这些产品，找出限制组装数量的**瓶颈组件**（库存量 / BOM需求量 最低的）
3. 安排打印该瓶颈组件，更新模拟库存
4. 重复步骤 2-3，直到生成足够的富余任务

### 7.3 富余任务上限

设定**目标富余产品数**（代码常量 `SURPLUS_TARGET_PRODUCTS = 5`），当模拟库存比基线多出该数量的可组装完整产品时停止生成富余任务。避免无限制生产导致某一组件堆积过多。

### 7.4 富余任务调度

- 富余任务同样遵守当前所选策略的选择规则
- 在甘特图和列表中以不同颜色/标签区分（橙色/"富余"标签）

---

## 8. 批次调度流程

```
1. 按订单 FIFO 顺序逐个计算需求（如有产品过滤则只考虑匹配的订单项）
   → 选择打印配置 → 得到需求任务池
2. （如开启富余）计算预计库存 → 瓶颈算法生成富余任务池（同样受产品过滤约束）
3. 根据所选策略初始化上下文：
   - product_first：构建产品单元队列 + 初始化模拟库存
   - utilization：无额外初始化
4. 初始化：所有打印机可用时间 = 排班开始时间
5. 循环：
   a. 找到最早可用的打印机时间
   b. 找到该时间之后最早的操作窗口启动点
   c. 如果启动点 >= 排班截止时间 → 结束
   d. 收集所有在此启动点可用的打印机
   e. 为每台可用打印机分配一个任务：
      - 优先从需求任务池取，取完后从富余任务池取
      - 跳过会超出排班截止时间的任务
      - 按所选策略的评分规则选择
   f. 更新各打印机的可用时间 = 任务结束时间 + 换料时间
   g. （product_first 策略）更新模拟库存，检查是否有产品可组装
   h. 如果本批次未分配任何任务 → 结束
6. 保存排班表
```

---

## 9. 同步策略（当前简化）

当前实现中，同一批次的所有打印机在同一时刻启动。未来可优化为：

- 允许子组同步：2 台先结束的打印机可以先启动新任务，不需等另外 2 台
- 约 10 分钟窗口内结束的打印机可以合并为一次操作

---

## 10. 订单与需求计算

### 10.1 需求计算

按 `created_at` 升序逐个处理待处理订单，维护一个滚动供给池：

1. 初始供给 = 当前库存 + 早于排班日的已排班产出
2. 对每个订单：
   - 计算该订单的组件需求（如有产品过滤，只计算匹配产品的需求）
   - 从供给池中扣除可满足的部分，得到净需求
   - 如果净需求为空 → 库存已满足此订单，跳过
   - 为净需求选择打印配置，生成任务
   - 任务产出中超出净需求的溢出量加回供给池

### 10.2 效果

- 库存已能满足的早期订单被跳过，不浪费产能
- 前序订单打印配置的溢出量自动供给后续订单，减少重复打印

## 11. 指定产品过滤

### 11.1 功能描述

用户可以在生成排班时手动指定一组产品（`target_product_ids`），本次排班将**仅考虑**这些产品的相关组件。

### 11.2 过滤效果

指定产品过滤后，以下环节均受影响：

| 环节 | 无过滤（默认） | 有过滤 |
|------|---------------|--------|
| **需求计算** | 所有待处理订单的所有产品 | 只计算匹配产品的订单项需求 |
| **任务池** | 所有需求组件 | 只包含匹配产品的组件 |
| **富余生产** | 所有有订单的产品 | 仅补充匹配产品的组件 |
| **产品凑齐评分** | 所有产品单元 | 仅评估匹配产品的完成度 |

### 11.3 与策略的组合

产品过滤与调度策略完全正交，可以自由组合：

| | 无过滤 | 过滤"纯白转角书桌" |
|---|---|---|
| **优先凑齐** | 按订单顺序逐个凑齐所有产品 | 只凑白色书桌，忽略鞋子等 |
| **最大化利用率** | 所有订单组件，最小空闲 | 只排白色书桌组件，最小空闲 |

### 11.4 无订单时的行为

如果指定了产品过滤，但当前没有该产品的待处理订单：
- 需求任务池为空（无订单需求）
- 如果开启了富余生产，仍然会生成该产品的富余任务
- 效果等价于"我就是想生产这个产品的库存"

> 这意味着"无视订单、直接指定生产某产品"无需单独实现——用户只需指定产品 + 开启富余生产即可达到相同效果。

### 11.5 共享组件

不同产品可能共享组件（如不同配色书桌共享"固定件"）。指定产品过滤时：
- 只计算被选中产品对共享组件的需求量
- 不会顺带为未选中产品生产共享组件
